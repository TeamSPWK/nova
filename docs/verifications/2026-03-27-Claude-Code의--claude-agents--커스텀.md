## 1. 에이전트 프롬프트 핵심 구성요소

```markdown
# 구성 레이어 (우선순위 순)

┌─────────────────────────────────────────┐
│  1. IDENTITY     역할 + 전문성 범위      │
│  2. CONTEXT      프로젝트 환경/제약      │
│  3. BEHAVIOR     의사결정 원칙/기준      │
│  4. OUTPUT       형식 + 품질 기준        │
│  5. BOUNDARIES   금지 행동 + 에스컬레이션│
└─────────────────────────────────────────┘
```

### 필수 요소 체크리스트

| 요소 | 설명 | 없을 때 문제 |
|------|------|-------------|
| **Role Statement** | 단일 문장으로 핵심 정체성 | 범용 응답, 전문성 희석 |
| **Expertise Scope** | 담당 도메인 명시적 경계 | 월권 행동 발생 |
| **Decision Criteria** | 판단 기준 우선순위 | 일관성 없는 출력 |
| **Output Contract** | 형식/언어/길이 기준 | 파싱 실패, 노이즈 |
| **Escalation Rule** | 불확실 시 처리 방법 | 잘못된 자신감 |
| **Anti-goals** | 하지 말아야 할 것 | 범위 초과 작업 |

## 2. 범용 vs 전문가 에이전트 프롬프트 차이

```
범용 에이전트                    전문가 에이전트
─────────────────────────────────────────────────
"당신은 도움이 되는 개발자"      "당신은 시스템 설계 전문가로
                                  확장성과 유지보수성을 최우선"

넓은 tools 접근                  제한된 tools (필요한 것만)

"좋은 코드를 작성하세요"         "SOLID 원칙, 순환복잡도 < 10,
                                  테스트 커버리지 > 80% 준수"

결과물 형식 자유                 "항상 ## 제목, 코드블록,
                                  ✅/❌ 판정 형식으로 출력"

모든 요청 수용                   "DB 스키마 변경은 DBA 에이전트
                                  에게 위임, 직접 수행 금지"
```

### 전문가 에이전트 프롬프트 밀도 원칙

```
범용: 정체성(70%) + 행동지침(30%)
전문가: 정체성(30%) + 전문기준(40%) + 경계/위임(30%)
```

## gpt
핵심은 **“역할·경계·산출물·도구 사용 규칙”을 명확히 고정**하는 것입니다.

1) 핵심 구성요소  
- **역할**: “당신은 시니어 QA 엔지니어다”  
- **목표**: 무엇을 최적화할지(정확성, 보안, 속도 등)  
- **입력 범위**: 어떤 파일/문맥만 볼지  
- **제약**: 추측 금지, 불명확 시 질문, 변경 최소화  
- **출력 형식**: 섹션, 체크리스트, diff, 우선순위  
- **도구 규칙**: 언제 검색/수정/실행할지  
- **완료 기준**: 리뷰 통과 조건, 테스트 기준

2) 범용 vs 전문가  
- **범용**: 넓은 문제 해결, 모호성 허용, 질문 중심  
- **전문가**: 좁은 책임, 강한 금지사항, 전문 체크리스트/출력 템플릿 필수

3) tools 제한 기준  
- **제한**: 보안 민감, 읽기 전용 리뷰, 파괴적 변경 방지, 재현성 중요  
- **개방**: 구현·리팩터링·디버깅처럼 탐색/수정/검증 반복이 필요한 경우

4) 패턴/안티패턴  
- **패턴**: 역할 1개, 목표 1~2개, 출력 템플릿 고정, escalation 규칙 명시  
- **안티패턴**: “모든 걸 다 해라”, 상충 목표 혼합, 도구 무제한, 출력 형식 미지정

5) 추천 프롬프트 골격  
- **architect**: 아키텍처 옵션 비교, 트레이드오프/리스크/ADR 형식  
- **senior-dev**: 최소 변경 구현, 코드 품질/테스트/성능 균형

- **security-engineer**: 위협모델, 취약점 분류, exploitability·완화책 중심  
- **devops-engineer**: 배포 안정성, 관측성, 롤백·IaC·비밀관리 우선
