## 질문
AXIS Kit은 범용 AI Engineering 방법론 프레임워크다. 현재 /team qa 프리셋은 정적 코드 분석 기반 QA만 수행한다. 실전에서 '화면 검증(Visual QA)을 못 했다'는 피드백이 반복되고 있다.

문제: Claude Code 에이전트 환경에서 실제 브라우저를 띄워 스크린샷을 찍는 것은 환경 의존적이다. Playwright 같은 도구가 프로젝트에 설치되어 있어야만 가능하다.

질문: AXIS Kit(범용 방법론 프레임워크)에서 Visual QA를 어떻게 다뤄야 하는가? 선택지는 3가지다:
A) /team qa에 Visual QA 에이전트 역할을 추가하되, Playwright가 없으면 정적 분석으로 폴백
B) 별도 /team visual-qa 프리셋을 만들어 Playwright 의존성을 명시
C) AXIS Kit 범위 밖으로 두고, 프로젝트별로 알아서 세팅하도록 가이드만 제공
어떤 접근이 가장 적절한가?

## claude
## 권장 접근: **A + B 조합 (계층적 설계)**

---

### 핵심 판단 근거

AXIS Kit이 "범용 방법론 프레임워크"라면, **역할 정의**는 프레임워크 책임이고 **환경 의존성 해결**은 프로젝트 책임입니다.

---

### 구체적 설계

**`/team qa` (A 적용)**
```
Visual QA 에이전트 역할을 명세에 포함
→ Playwright 감지 시: 실제 스크린샷 검증 수행
→ 미감지 시: CSS/HTML 정적 분석 + "Visual QA 미수행" 명시적 경고 출력
```

**`/team visual-qa` (B 적용)**
```
Playwright 필수 의존성 명시
전용 워크플로우: 해상도별 검증, 컴포넌트 스냅샷 비교 등
```

---

### C를 단독 선택하면 안 되는 이유

> "피드백이 반복된다" = 이미 프레임워크 책임 영역

가이드만 제공하면 팀마다 구현이 달라져 **방법론의 일관성이 깨집니다.**

---

### 결론

| 선택지 | 역할 | 의존성 |
|--------|------|--------|
| `/team qa` | Visual QA 역할 **정의** | 폴백으로 포용 |
| `/team visual-qa` | Visual QA **전문 실행** | 명시적 요구 |

**역할 명세는 프레임워크가 소유하고, 실행 가능 여부는 환경이 결정하는 구조**가 범용 프레임워크의 올바른 설계 원칙입니다.

## gpt
**권장: B + A 일부 흡수**가 가장 적절합니다.

- **B가 적합한 이유**
  - AXIS Kit은 **범용 프레임워크**이므로, 환경 의존적인 브라우저 실행을 `/team qa` 기본책임에 넣으면 일관성이 깨집니다.
  - Visual QA는 **도구/런타임 의존성**이 강해 별도 프리셋으로 분리해야 기대치 관리가 명확합니다.
  - `/team visual-qa`에서 **Playwright 설치, 브라우저 실행, 스크린샷 비교**를 명시하면 실전 활용성이 높습니다.

- **A를 기본으로 삼기 어려운 이유**
  - 폴백이 있으면 사용자는 “QA가 됐다”고 오해하기 쉽지만, 실제론 **시각 검증이 수행되지 않았을 수 있음**.
  - 책임 경계가 흐려집니다.

- **C만 선택하기 아쉬운 이유**
  - 범용 프레임워크라도 **표준 확장 경로**는 제공해야 합니다.
  - 완전 위임하면 반복 피드백 문제가 계속됩니다.

**결론:**  
`/team qa`는 정적/기본 QA 유지, 대신 **“Visual QA는 별도 `/team visual-qa` 사용”**으로 분리하세요. `/team qa`는 필요 시 **visual-qa 연계 안내만** 제공하는 것이 최선입니다.

## gemini
소프트웨어 아키텍처 관점에서 **B) 별도 /team visual-qa 프리셋을 신설하는 방식**이 가장 적절합니다.

**선택 이유:**

1.  **관심사 분리(Separation of Concerns):** 정적 코드 분석과 런타임 기반의 화면 검증은 실행 환경, 리소스 요구사항, 피드백의 성격이 본질적으로 다릅니다. 이를 분리해야 각 에이전트의 프롬프트를 전문화할 수 있습니다.
2.  **명시적 의존성 관리:** 범용 프레임워크에서 '마법' 같은 폴백(Option A)은 사용자에게 혼란을 줍니다. 시각적 검증에는 Playwright 같은 도구가 필수임을 명시하고, 해당 프리셋 실행 시 환경을 체크하도록 설계하는 것이 유지보수에 유리합니다.
3.  **확장성 및 유연성:** 향후 시각적 회귀 테스트(VRT)나 접근성 검사 등 화면 중심의 복잡한 워크플로우를 추가할 때 기존 정적 분석 로직과의 충돌 없이 독립적인 고도화가 가능합니다.

**결론:** AXIS Kit의 정체성인 '방법론'을 유지하되, 환경 의존적인 기능은 전용 프리셋으로 격리하여 사용자가 필요할 때 선택적으로 환경을 구축하고 사용하게 유도하는 것이 아키텍처적으로 가장 견고합니다.
