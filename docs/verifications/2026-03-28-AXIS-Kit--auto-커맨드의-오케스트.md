# AXIS Kit /auto 오케스트레이션 설계

## 1) Plan → Design: 순차 실행 권장

**결론: 순차 실행이 맞다.**

`/auto`의 핵심 가치는 **품질 보장**이지 속도가 아님. Plan이 Design의 입력 계약(interface contract) 역할을 해야 하므로, Plan 없이 Design을 시작하면 재작업 비용이 병렬화 이득을 초과함.

단, Plan 완료 후 Design 시작 전 **자동 정합성 체크(Plan→Design handoff validation)** 삽입 권장.

---

## 2) Agent Teams 활용 기준

| 조건 | 전략 |
|------|------|
| 태스크 간 의존성 낮음 + 파일 3개↑ | `/team` 병렬 |
| 도메인 경계가 명확 (BE/FE/DB 분리) | `/team` 전문화 |
| 단일 모듈, 의존성 체인 강함 | 단일 에이전트 |

**판단 기준: Plan 단계 산출물의 컴포넌트 수 ≥ 3이면 `/team` 트리거.**

---

## 3) 자동 수정 vs 사용자 보고

**하이브리드 전략:**

```
Gap/Review 발견
  ├─ Severity: LOW  → 자동 수정 → 재검증 (최대 2회)
  ├─ Severity: MED  → 자동 수정 → 재검증 (최대 1회) → 실패시 보고
  └─ Severity: HIGH → 즉시 사용자 보고 (자동 수정 금지)
```

**최대 반복: 전체 3회 cap.** 초과 시 강제 중단 + 현재 상태 보고. 무한루프 방지가 최우선.

## 핵심 설계 원칙

> **`/auto`는 "자율 실행"이 아니라 "감독된 자동화"여야 한다.**
> HIGH severity에서 자동 수정을 허용하는 순간 시스템 신뢰도가 붕괴됨.
> 반복 횟수 cap과 severity 게이트가 `/auto`의 안전핀.

## gpt
권장안은 **기본 순차 + 조건부 병렬/팀 + 제한적 자동수정**입니다.

1) **Plan/Design**
- 기본은 **순차(Plan→Design)** 권장.
- 이유: /auto는 품질·정합성이 핵심이라 Design이 Plan 제약/범위를 반드시 상속해야 함.
- 단, 소규모·반복 작업은 **병렬 초안 생성 후 정합성 머지** 가능.
- 기준: 불확실성 높음/요구 복잡/의존성 많음 → 순차, 단순 CRUD/템플릿성 작업 → 병렬 허용.

2) **구현 단계**
- 기본은 **단일 에이전트**.
- **/team 사용 기준**: 다중 모듈, 프론트/백/인프라 분리, 병렬화 이득 큼, 컨텍스트가 1 에이전트 한계 초과.
- 작은 변경, 강한 일관성 필요, 충돌 위험 높음 → 단일 에이전트.

3) **/gap, /review 후 처리**
- **저위험·국소 수정은 자동 수정 후 재검증**, 구조 변경/요구 변경/애매한 판단은 사용자 보고.
- 최대 반복은 **2회** 권장(3회 이상은 수렴보다 루프 위험 큼).
- 종료 조건: 동일 오류 재발, 품질 개선 미미, 범위 확장 발생.

4) **보고 형식**
- 실용 포맷: **요약 + 단계별 결과 + 이슈/결정 + 최종 상태**.
- 예:
  - 목표/범위
  - Plan 요약
  - Design 요약
  - 구현 내역
  - Gap/Review 결과
  - 자동 수정 여부 및 반복 횟수
  - 남은 리스크/사용자 확인 필요사항
  - 최종 판정: 완료/부분완료/중단

## gemini
소프트웨어 아키텍처 전문가로서 `/auto` 커맨드 설계를 위한 최적의 오케스트레이션 전략을 제안합니다.

1. **Plan & Design: 순차 생성(Sequential)**
아키텍처 정합성이 최우선입니다. Plan이 Design의 가이드라인(Constraints) 역할을 수행해야 구현 단계의 정밀도가 보장됩니다. 속도보다 '재작업 비용 감소'에 초점을 맞춰 순차 실행을 권장합니다.

2. **구현: Agent Teams 활용**
복잡한 비즈니스 로직과 파일 간 의존성을 다루는 `/auto` 특성상 **Agent Teams**가 필수입니다. (기준: 수정 파일 3개 이상 또는 API 계약 변경 포함 시). 역할 분담(작성/검증)을 통해 단일 에이전트의 컨텍스트 한계를 극복해야 합니다.

3. **오류 처리: 2회 자동 수정 후 보고**
단순 문법 오류나 의존성 누락은 **최대 2회** 자동 수정을 시도합니다. 3회 이상 실패는 설계 결함일 가능성이 높으므로, 즉시 사용자에게 보고하여 리소스 낭비를 막고 인간의 의사결정을 유도합니다.

4. **보고 형식: Markdown 요약 대시보드**
단계별 성공 여부, 주요 변경 사항, 잔존 리스크를 한눈에 파악할 수 있는 **Markdown 대시보드**가 가장 실용적입니다. 상세 로그는 `details` 태그로 숨겨 가독성을 확보하십시오.
