## 질문
【맥락】
Nova는 Claude Code 플러그인으로 배포되는 AI Agent Ops 프레임워크. 5기둥(환경·맥락·품질·협업·진화)으로 구성됨. 기존 스킬: context-chain, deepplan, evaluator, evolution, field-test, jury, orchestrator, ux-audit, worktree-setup. 기존 파이프라인은 /nova:plan → /nova:design → 구현 → /nova:run(구현+검증) → /nova:review/check. deepplan은 Explorer×3(code/risk/option) 병렬 → Synthesizer → Critic → Refiner 구조로 Plan에 Risk Map + Unknowns + Verification Hooks를 채운다. design은 Sprint Contract(사전 합의 Done 조건)를 요구한다.

【문제 가설】
Nova 전 파이프라인은 "무엇을 만들지는 이미 결정됐다"는 전제를 둔다. 즉 제품의 Desirability(사용자가 진짜 원하는가) 및 핵심 가정의 실험적 검증 단계가 빠져있다. Lean Startup의 RAT(Riskiest Assumption Test), XP의 Spike, Design Thinking의 3차원(Desirability/Feasibility/Viability) 같이 "가장 위험한 가정을 가장 싸게 실험으로 먼저 검증한다" 개념이 어디에도 없다. deepplan의 risk-explorer는 실패 시나리오를 나열만 할 뿐, "어느 가정이 자명하고 어느 가정을 반드시 실험으로 쳐봐야 하는지" 판정하지 않는다.

【제안】
Plan 단계 앞에 /nova:spike (또는 /nova:poc) 스킬을 신설한다. 6단계:
① 가정 추출 → ② 자명성 판정 표(자명/검증필요) → ③ TOP-3 가장 위험한 가정 선정 → ④ 각각에 정량 성공 기준(예: 10회 중 7회 성공, N명 중 M명 이해) → ⑤ 최소시간(하루~이틀) PoC 범위 → ⑥ go/no-go 판단. API/코드/사용자 실험 무관하게 "가장 싼 실험"을 선택한다. 결과가 PASS면 /nova:plan으로, FAIL이면 아이디어 자체 포기 또는 피봇.

【질문】
이 /nova:spike 스킬을 Nova에 추가할 가치가 있는가? 3가지 관점에서 판단해라:

(1) **실제 수요**: Claude Code 플러그인 사용자(주로 개인 개발자/작은 팀)가 "만들 가치 있는지"부터 의심하는 빈도가 높은가, 아니면 대부분 이미 만들기로 한 것을 잘 만드는 데 관심이 있는가? 즉, 이 스킬의 호출 빈도가 /nova:plan 대비 의미 있게 나올 것인가?

(2) **기존 스킬과의 중복**: deepplan의 risk-explorer + option-explorer로 이미 충분한가? /nova:spike가 deepplan 안의 한 Phase로 흡수되는 것이 더 낫지 않은가? 독립 스킬로 분리해야 할 명확한 이유가 있는가?

(3) **Nova 철학 정합성**: Nova는 AI가 독단적으로 쓰기보다 Claude Code 세션 안에서 실행된다. "6단계 PoC 프로세스"가 LLM 주도로 의미 있게 돌아갈 수 있는가, 아니면 결국 사용자가 실제 실험(친구 5명에게 물어보기 등)을 수행해야 해서 스킬이 빈 껍데기가 될 위험은 없는가?

각 관점에서 "도입해야 한다 / 보류해야 한다 / deepplan에 흡수" 중 하나로 결론을 명시하고 근거를 대라.

## claude
# /nova:spike 도입 타당성 분석

## (1) 실제 수요 → **보류**

Claude Code 사용자 패턴상 "이미 만들기로 결정한 것"을 가져오는 비율이 압도적으로 높다. 코드 편집기 플러그인이라는 맥락 자체가 **실행 모드** 진입 이후다. "만들 가치 있나?"는 Notion/Figma/대화 단계에서 이미 소화된다. `/nova:spike` 호출 빈도가 `/nova:plan`의 10~15% 미만일 가능성이 높아 생태계 내 마찰만 늘린다.

---

## (2) 기존 스킬 중복 → **deepplan에 흡수**

현 risk-explorer의 결함은 **"나열만 하고 판정하지 않는다"**는 것이다. 이는 독립 스킬 신설이 아니라 deepplan Critic 페이즈에 **Assumption Triage** 스텝 추가로 해결 가능하다.

```
Critic → [기존 실패 시나리오 비판]
       + [가정 자명성 판정표: 자명/검증필요]  ← 추가
       + [TOP-3 RAT 선정 + 정량 기준]        ← 추가
```

Refiner가 이를 받아 Plan에 "실험 선행 조건"으로 명시하면 충분하다. 별도 스킬은 파이프라인 진입점 분산만 야기한다.

---

## (3) Nova 철학 정합성 → **보류**

가장 치명적 문제다. 6단계 중 **④~⑥(정량 기준·실험 수행·go/no-go)**은 LLM이 *설계*할 수 있지만 *실행*할 수 없다. "친구 5명에게 물어보기", "API 실제 호출 레이턴시 측정"은 세션 외부 행위다. 결과적으로 스킬이 **실험 설계서 생성기**로 퇴화하며, 이는 deepplan 산출물과 차별성이 없다.

---

## gpt
결론은 **조건부 도입**입니다.

1) **실제 수요: 보류해야 한다**  
Claude Code 주사용자는 대체로 “무엇을 만들지”보다 “빨리/안전하게 구현” 니즈가 강합니다. 따라서 `/nova:plan`급 호출 빈도는 어렵습니다. 다만 0→1, 사이드프로젝트, 신규 기능 탐색에선 분명 수요가 있어 **저빈도·고가치** 스킬입니다.

2) **기존 스킬과의 중복: 독립보다 deepplan에 흡수**  
deepplan은 현재 리스크를 “설명”하지 “실험 설계”하진 않습니다. 그래서 개념상 공백은 맞습니다. 하지만 별도 `/spike`는 파이프라인 분기와 학습비용을 늘립니다. 우선 **deepplan의 선행 Phase로 RAT/Spike 모드**를 넣는 게 적절합니다. 반복 사용이 확인되면 분리하세요.

3) **Nova 철학 정합성: 도입해야 한다(단, 인간 실행 전제)**  
LLM이 실험 대상자 역할을 대체하진 못해도, 가정 추출·우선순위·성공기준·최소실험 설계는 매우 잘합니다. 즉 “실험 수행”이 아니라 **실험 설계와 go/no-go 문서화**에 가치가 있습니다. 빈 껍데기 방지를 위해 출력물을 `Assumption Ledger + Test Card + Decision Gate`로 고정하세요.

## gemini
소프트웨어 아키텍처 전문가로서 `/nova:spike` 도입에 대한 의견을 제시합니다.

(1) **실제 수요: 도입해야 한다.**
개인 개발자와 소규모 팀에게 가장 비싼 비용은 '잘못된 것을 만드는 시간'입니다. Claude Code 사용자는 빠른 프로토토이핑을 원하므로, 단순 구현 전 "이 가설이 유효한가?"를 묻는 요구는 잠재적으로 매우 높습니다. `/nova:plan`이 '어떻게'에 집중한다면, `spike`는 '왜'와 '생존'을 다루어 자원 낭비를 막는 강력한 진입점이 될 것입니다.

(2) **기존 스킬과의 중복: 독립 스킬로 분리.**
`deepplan`은 실행 가능한 설계를 목표로 하기에 이미 '구현'을 전제합니다. 반면 `spike`는 '기각(No-Go)'을 목표로 하는 과학적 실험 단계입니다. 실패를 장려하는 실험 단계가 `deepplan`에 흡수되면 설계 복잡도만 높이고 본질이 희석됩니다. 파이프라인의 가장 앞단에서 의사결정을 지원하는 독립 스킬로서 정체성이 명확합니다.

(3) **Nova 철학 정합성: 도입해야 한다.**
Nova는 단순 자동화가 아닌 AI와 인간의 '협업 및 진화'를 지향합니다. LLM이 실험 프로토콜(성공 기준 등)을 설계하고 사용자가 수행 결과를 피드백하는 구조는 Nova의 5기둥 중 '진화'와 '협업'에 완벽히 부합합니다. 이는 단순 코드 생성을 넘어 제품 전략을 가이드하는 에이전트로 Nova를 격상시킬 것입니다.

**결론:** `/nova:spike`는 Nova를 단순 도구에서 비즈니스 가치를 검증하는 **Product Strategy Agent**로 진화시키는 핵심 스킬이 될 것입니다. 바로 도입하십시오.

## 합의 분석
- **합의율**: 60%
- **판정**: human_review
- **요약**: 가정 검증 및 실험 설계 로직의 필요성에는 합의가 이루어졌으나, 독립 스킬 신설보다는 deepplan 고도화 의견이 우세하며 실제 사용자 수요에 따른 전략적 판단이 필요함

### 공통점
- 현재 deepplan의 risk-explorer가 리스크를 나열만 할 뿐 판정/실험 설계 기능이 부족하다는 점에 동의함
- LLM이 실험의 설계(가정 추출, 성공 기준 수립) 단계에서는 충분한 역할을 할 수 있음을 인정함
- 실제 실험 수행은 AI가 아닌 사용자(인간)의 영역이 포함될 수밖에 없다는 한계를 공통적으로 지적함

### 차이점
- 구현 형태: 독립 스킬(/nova:spike)로 분리해야 한다는 의견(Gemini)과 deepplan의 내부 페이즈로 흡수해야 한다는 의견(Claude, GPT)이 대립함
- 수요 예측: 코드 편집기 맥락상 수요가 낮을 것이라는 비관론(Claude, GPT)과 잠재적 가치가 매우 높다는 낙관론(Gemini)이 존재함
- 철학적 가치: 단순한 '문서 생성기'로 전락할 위험성(Claude) vs 제품 전략 에이전트로의 진화(Gemini)라는 시각 차이
