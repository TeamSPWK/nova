# X-Verification: Nova v2.4.2 아키텍처와 한계 냉정 평가

> 날짜: 2026-03-31
> 합의 수준: Strong Consensus
> AI: architect, senior-dev, qa (에이전트 모드)

## 질문

Nova 프레임워크(v2.4.2)의 현재 아키텍처와 한계를 냉정하게 평가해줘.

### 핵심 질문
1. Generator-Evaluator 분리는 진짜 독립 검증인가, 자기 대화인가?
2. 컨텍스트 피로 문제를 Nova가 실제로 해결하고 있는가?
3. Paperclip 같은 외부 오케스트레이터와 비교한 진짜 강점과 한계는?
4. Nova가 풀고 있는 문제 vs 풀지 못하는 문제 MECE 분류
5. v3.0으로 가려면 반드시 해결해야 할 구조적 약점은?

### 맥락
- 제작자 핵심 동기: "단일 에이전트로 할수록 메인 에이전트가 피로해지면서 수행능력이 떨어진다"
- Paperclip(외부 멀티에이전트 오케스트레이터)과의 비교가 계기

## 아키텍트

Nova v2.4.2는 "프롬프트 기반 방법론 프레임워크"다. 런타임 격리나 상태 관리 메커니즘이 없다.

- **Generator-Evaluator 분리**: 구조적 독립성 없음. "적대적 자세"는 프롬프트 지침일 뿐, 물리적 격리가 아니다.
- **컨텍스트 피로**: nova-context-chain이 세션 간 단절은 완화하지만, 단일 세션 내 컨텍스트 팽창은 막지 못한다. 피로해진 에이전트가 자신이 피로하다는 것을 정확히 판단할 수 없다.
- **v3.0 핵심**: ①검증 강제화 런타임 훅 부재 ②컨텍스트 스케줄러 없음 ③/xv의 외부 API가 기본값이 아닌 선택 옵션

## 시니어 개발자

판정: ⚠️ NEEDS WORK

- /xv 결과, /gap 출력, /review 리포트가 메인 스레드로 돌아오면 결국 같은 윈도우를 채운다. 증상 완화지 근치가 아니다.
- Nova의 진짜 강점은 "설치 마찰 제로". 약점은 "진짜 격리가 없다"는 것.
- "개인 개발자의 워크플로우 규율 도구"로는 완성도 높음. 에이전트 격리를 Paperclip 수준으로 포장하면 기술 부채.
- /xv.md 19번째 줄의 경고문이 Nova 아키텍처의 정직한 자기평가 — v3.0 설계의 출발점.

## QA/보안

- "독립 검증"은 구조적 허구. 프로세스 분리지 모델 분리가 아니다.
- 118개 테스트 전부 마크다운 파일 존재 여부 확인. AI 판단 품질의 회귀를 잡을 수 없다.
- Evaluator False Negative가 발생해도 CI 통과 — 자동 탐지 불가.
- 가장 정직한 설명: "같은 모델이 다른 관점 프롬프트로 자기 작업을 재검토하는 구조화된 습관 강제 도구"

## 합의 분석

- **합의 수준**: Strong Consensus
- **요약**: Nova는 "워크플로우 규율 도구"로는 유효하나, "독립 멀티에이전트 검증"은 구조적 허구

### 공통점
- Generator-Evaluator 분리는 프롬프트 역할극이지, 물리적 격리가 아니다
- 컨텍스트 피로 문제를 완화할 뿐 해결하지 못한다
- /xv Mode B는 동일 모델 편향 — "교차검증"이라는 명칭이 과장
- Nova의 진짜 가치는 "개발 규율 강제 + 설치 마찰 제로"
- v3.0 핵심은 외부 모델 연동 + 진짜 세션 격리

### 차이점
- 아키텍트: 검증 강제화 런타임 훅, 컨텍스트 스케줄러 필요 강조
- 시니어 개발자: 포지셔닝 정직성 — Paperclip 수준으로 포장하지 말라
- QA/보안: 테스트가 AI 동작이 아닌 파일 존재만 검증하는 점 지적
