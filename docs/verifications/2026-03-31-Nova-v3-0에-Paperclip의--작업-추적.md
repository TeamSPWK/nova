## 배경
- Nova는 Claude Code 플러그인(Quality Gate) — 검증에 집중
- 오케스트레이션은 Claude Code Agent Teams가 담당

## 제안 1: 작업 추적 (Lightweight Task Registry)
docs/tasks/ 또는 docs/registry.md에 에이전트별 작업 상태를 기록:
- 누가(에이전트), 무엇을(작업), 언제, 상태(진행중/완료/블로킹)
- /nova:next가 이걸 읽어서 더 정확한 추천을 할 수 있음
- Agent Teams의 각 에이전트가 작업 시작/완료 시 자동 기록

## 제안 2: 감사 로그 (Audit Trail)
docs/audit-log.md에 검증 결과 + 작업 이력을 시간순 기록:
- Quality Gate 판정 (PASS/FAIL/CONDITIONAL)
- 누가 어떤 파일을 변경했는지
- /xv 교차검증 결과 요약
- 이미 docs/verifications/에 저장하는 것의 확장

## 제안 3: /nova:init의 가치 강화  
- 프로젝트 분석 → 도메인 특화 에이전트 설계 → .claude/agents/ 생성
- Agent Teams와 자연스럽게 연동 (생성된 에이전트를 Teams가 바로 사용)
- Paperclip의 '조직도' 개념을 에이전트 파일 내 reportsTo 같은 메타데이터로 경량 구현

# Nova v3.0 경량 Paperclip 통합 평가

## 1. Quality Gate 정체성과 충돌 여부

**충돌 없음 — 단, 역할 경계 명확화 필수**

세 제안 모두 Nova의 검증 기능을 *강화*하는 방향이나, 위험 지점이 있습니다.

| 제안 | 판단 | 주의사항 |
|------|------|----------|
| 작업 추적 | ✅ 보완적 | Nova가 **기록자**가 되면 안 됨. 에이전트가 직접 기록해야 함 |
| 감사 로그 | ✅ 핵심 강화 | 이미 verifications/와 자연스럽게 연결됨 |
| /nova:init 확장 | ⚠️ 경계 모호 | 오케스트레이션 영역 침범 위험. "설계 지원"에 한정해야 함 |

## 2. 마크다운 파일 관리의 실용성과 한계

**실용적 — Claude Code 환경에서 오히려 최적**

```
✅ 장점
- 별도 인프라 불필요 (DB, 서버 없음)
- 에이전트가 직접 읽기/쓰기 가능
- Git으로 변경 이력 자동 추적
- 사람이 직접 검토 가능

⚠️ 한계 (규모 임계점)
- 동시 쓰기 충돌: 에이전트 3개 이상이 동시 기록 시 race condition
- 검색 성능: 작업 100개+ 넘으면 /nova:next 컨텍스트 로딩 부담
- 파싱 의존성: 마크다운 구조 변경 시 전체 에이전트 동작 깨짐
```

**권장 구조:**
```markdown
<!-- docs/registry.md — 엄격한 스키마 준수 필수 -->
| agent | task_id | status | started | blocked_by |
|-------|---------|--------|---------|------------|
```
스키마를 Nova가 검증하는 것도 Quality Gate 역할에 자연스럽게 부합.

## 3. /nova:init이 Paperclip 조직도를 대체할 수 있는가?

**부분 대체 가능 — 단 "관리"가 아닌 "선언"에 집중해야**

```yaml
# .claude/agents/backend-reviewer.md 예시
---
role: backend-reviewer
reportsTo: lead-orchestrator
scope: [src/api/, src/db/]
escalateTo: nova-quality-gate
handoffTo: [frontend-reviewer, test-agent]
---
```

Paperclip 조직도와의 차이:
- Paperclip: **런타임 라우팅** (누가 다음 작업 받을지 동적 결정)
- Nova 방식: **정적 선언** (에이전트가 선언을 읽고 스스로 판단)

→ Claude Code Agent Teams가 동적 라우팅을 담당하므로, Nova는 정적 메타데이터만 제공하면 충분. **역할 분리가 오히려 깔끔함.**

## gpt
평가: **충돌은 없고, “검증 보조 메타레이어”로 제한하면 정체성 유지 가능**합니다.

1. **정체성 충돌**  
- 충돌 없음. 단, Nova가 **오케스트레이터가 아니라 증거 수집·판정 보조자**로 남아야 함.  
- 작업 추적/감사 로그는 추천 정확도와 검증 신뢰성 강화 용도로만.

2. **Markdown 실용성/한계**  
- 장점: 구현 빠름, Git diff 친화적, 인간/에이전트 모두 읽기 쉬움.  
- 한계: 동시성 충돌, 포맷 드리프트, 대규모 이력 검색/집계 약함.  
- 권장: **append-only + YAML frontmatter/고정 스키마**. 장기적으론 JSONL 병행.

3. **/nova:init vs 조직도**  
- 상당 부분 대체 가능. `role`, `responsibilities`, `reportsTo`, `handoffTo` 메타데이터면 충분.  
- 단, Paperclip식 조직 운영 전체는 아니고 **경량 책임 그래프** 수준이 적절.

4. **컨텍스트 피로 방지**  
- 원문 전체 주입 금지, **요약/최근 N개/현재 에이전트 관련 항목만** 읽기.  
- 상태 변경 시만 기록, 중복 로그 압축, `/nova:next`는 summary 우선.

5. **우선순위**  
① **감사 로그**(기존 verifications 확장이라 ROI 최고)  
② **작업 추적**(`/nova:next` 품질 향상)  
③ **/nova:init 메타데이터 강화**(효과 크지만 설계 비용 높음)

## gemini
아키텍처 전문가로서 제안하신 Nova v3.0의 방향성을 평가합니다.

1. **정체성:** 충돌하지 않습니다. 오히려 '무엇을' 검증했는지에 '누가, 왜'라는 맥락을 더해 Quality Gate의 판단 근거를 강화합니다.
2. **실용성:** 마크다운은 LLM 친화적이라 매우 실용적입니다. 단, 파일이 비대해지면 토큰 낭비가 발생하므로, 주기적인 요약(Summarization)과 아카이빙 전략이 병행되어야 합니다.
3. **조직도:** 가능합니다. 중앙 집중식 조직도보다 에이전트 메타데이터(`reportsTo`, `scope`) 방식이 에이전트 팀의 동적 확장성에 더 적합합니다.
4. **컨텍스트 피로 방지:** LLM이 전체 로그를 읽지 않게 하십시오. 현재 작업 중인 상태(`status.md`)만 주입하고, 상세 이력은 필요시에만 조회하는 'On-demand' 방식으로 설계해야 합니다.
5. **우선순위:** 
   - **1순위 (제안 1):** 실시간 협업 효율을 위한 작업 추적.
   - **2순위 (제안 3):** 체계적인 팀 구성을 위한 `/nova:init` 강화.
   - **3순위 (제안 2):** 사후 분석을 위한 감사 로그.

### 공통점
- Nova의 Quality Gate 정체성과 충돌하지 않으며, 오히려 검증의 맥락과 신뢰성을 강화함

- 에이전트 파일 내 메타데이터(reportsTo, scope 등)를 통한 경량 조직도 구현이 가능하고 효율적임
- 컨텍스트 피로를 막기 위해 전체 로그 주입 대신 요약(Summary) 및 필요 시 조회(On-demand) 방식을 채택해야 함

### 차이점
- 구현 우선순위: GPT는 감사 로그를 1순위로 꼽았으나, Gemini는 작업 추적을 1순위로 제안함
- 기술적 세부 구현: Claude는 엄격한 스키마 준수를 강조했고, GPT는 장기적으로 JSONL 병행을 제안함
