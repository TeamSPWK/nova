## 2. 자동 호출 잡음 위험
**중간 위험**. 휴리스틱 폭이 넓어 `styles/` 한 줄 수정에도 3인 감사가 돈다. 완화책:

- **첫 트리거 시 1회 안내 + 옵트아웃 유도**(깜짝 방지)
- `.nova/last-audit` 해시로 동일 변경 재감사 skip

## 3. 평가자 정체성
항목 A(일관성 붕괴=학습 부담)는 Cognitive Load 정체성과 **정합**. 항목 B(토큰 우회)는 엄밀히는 **유지보수성/DS 거버넌스**라 경계선. 그러나 6번째 평가자 폐기 원칙을 지키려면 흡수가 최선. → **B는 "하드코딩이 인지 일관성을 깨는 경우"로 문구 재정의**하면 정체성 유지.

## 4. 우선순위
**변경2 먼저, 변경1 나중**. 이유: 감사 품질이 먼저 탄탄해야 자동 호출이 가치 있음. 반대면 빈약한 감사가 자주 호출되어 신뢰 손상. 하나만 한다면 **변경2**.
