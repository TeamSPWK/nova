# AXIS Engineering v1.0

> AI 시대의 소프트웨어 개발 방법론
> **A**daptive · **X**-Verification · **I**dempotent · **S**tructured

---

## 1. 철학

**"처음부터 제대로 만들어 재작업을 제거하고, 매번 더 빠르게 만든다. 일관성이 속도를 만들고, 좋은 혁신은 즉시 흡수한다."**

기존 하네스 엔지니어링의 멱등성 철학을 계승하되, 정적인 규칙 체계를 **살아있는 시스템**으로 진화시킨다. AI는 도구가 아니라 팀원이며, 인간과 AI의 역할 경계를 명확히 하여 각자의 강점을 극대화한다. 궁극적 목표는 **생산성** — 같은 시간에 더 높은 품질의 결과물을 더 많이 만드는 것이다.

---

## 2. 생산성의 진짜 병목

AI 도구는 코드 생성 속도를 극적으로 높였다. 그러나 **빠르게 만드는 것**과 **빠르게 완성하는 것**은 다르다. 진짜 병목은 코드 작성이 아니라 그 전후에 있다.

| 병목 | 증상 | AXIS 해결책 |
|------|------|-------------|
| **방향 오류** | 잘못된 전제로 빠르게 구현 → 전면 재작업 | **Structured** — CPS로 문제를 먼저 정의, MECE로 빠짐없이 분해 |
| **AI 환각 누적** | 한 모델의 오류가 다음 작업으로 전파 → 디버깅 지옥 | **X-Verification** — 교차 검증으로 환각을 조기 차단 |
| **암묵지 소실** | 세션/팀원 교체 시 맥락 유실 → 동일 실수 반복 | **Adaptive + Context Chain** — 규칙으로 암묵지를 명시화, 메모리로 맥락 유지 |
| **설계-구현 괴리** | 설계와 코드가 따로 놀아 → 품질 저하, 유지보수 불가 | **X-Verification** — 역방향 검증으로 괴리를 자동 탐지 |

AXIS의 모든 구조는 이 네 가지 병목을 제거하기 위해 존재한다.

## 3. 핵심 원칙 (4 Pillars)

### Pillar 1: Structured — 구조가 품질을 만든다

> **생산성 기여**: 방향 오류를 원천 차단하여 재작업을 제거한다.

하네스 엔지니어링의 기반을 그대로 채택한다.

| 요소 | 설명 |
|------|------|
| CPS 프레임워크 | Context → Problem → Solution 순서로 사고 구조화 |
| MECE 분석 | 요구사항을 겹침 없이, 빠짐 없이 분해 |
| 문서 산출물 | 역기획 포함, 설계→구현 추적 가능한 구조화된 문서 |
| 린터 강제화 | 컨벤션을 사람의 의지가 아닌 시스템이 강제 |
| Eval 지표 | 조직 맞춤 품질 기준으로 측정 가능한 결과물 |

### Pillar 2: Idempotent — 누가 해도 같은 결과

> **생산성 기여**: 일관성이 속도를 만든다 — 예측 가능한 결과물은 리뷰와 통합 시간을 단축한다.

```
입력(요구사항) + 하네스(규칙 체계) = 일관된 결과물
```

- **모델 독립성**: Claude, GPT, Gemini 어떤 AI를 사용해도 동일 품질
- **사람 독립성**: 시니어든 주니어든 하네스를 따르면 기준선 보장
- **시간 독립성**: 6개월 뒤 돌아와도 읽고 유지보수 가능

### Pillar 3: X-Verification — 교차 검증으로 신뢰 확보

> **생산성 기여**: AI 환각과 설계-구현 괴리를 조기에 잡아 후반부 디버깅 비용을 제거한다.

단일 AI의 판단을 맹신하지 않는다.

**3단계 검증 체계:**

```
Level 1: Self-Review     — 생성한 AI가 스스로 검토
Level 2: Cross-Review    — 다른 AI 모델이 교차 검증
Level 3: Human-Review    — 사람이 최종 판단
```

**합의 프로토콜:**
- 합의율 90% 이상 → 자동 승인
- 합의율 70~89% → AI가 차이점 요약 후 사람이 판단
- 합의율 70% 미만 → 사람 필수 개입, 요구사항 재정의 검토

**Generator-Evaluator 분리 원칙:**
- 구현하는 AI(Generator)와 검증하는 AI(Evaluator)는 독립적으로 동작
- 자기 평가 편향을 제거: "모델은 자기 결과물을 칭찬하는 경향이 있다"
- 설계 문서의 "검증 계약"이 Generator와 Evaluator 사이의 합의 기준

**역방향 검증 (Bidirectional Traceability):**
- 정방향: 설계 → 구현 (의도대로 만들었는가?)
- 역방향: 구현 → 설계 (구현이 설계를 벗어나지 않았는가?)
- 갭 탐지 시 자동 리포트 생성 → PDCA 루프 진입
- 개선 추세면 계속, 정체되면 접근 전환 (Iterative Refinement vs Pivot)

### Pillar 4: Adaptive — 규칙이 스스로 진화한다

> **생산성 기여**: 좋은 혁신을 즉시 흡수하여 매번 더 빠르게 만든다 — 암묵지를 규칙으로 전환해 반복 실수를 없앤다.

하네스는 고정이 아니다. 프로젝트와 함께 성장한다.

**진화 사이클:**

```
감지(Detect) → 제안(Propose) → 승인(Approve) → 적용(Apply) → 검증(Verify)
```

1. **감지**: AI가 반복 패턴 3회 이상 탐지
2. **제안**: "이 패턴을 컨벤션으로 등록할까요?" 자동 제안
3. **승인**: 사람이 검토 후 승인/거부/수정 (AI가 독단적으로 규칙 변경 불가)
4. **적용**: 승인된 규칙이 린터/CLAUDE.md/설계 문서에 자동 반영
5. **검증**: 새 규칙 적용 후 기존 코드와의 충돌 여부 자동 확인

**변경 관리:**
- 모든 규칙 변경은 버전 관리 (git 추적)
- 롤백 가능: 규칙 변경이 품질 저하를 유발하면 즉시 되돌리기
- 변경 이력 대시보드로 "왜 이 규칙이 생겼는지" 추적 가능

## 4. 인간-AI 역할 경계 (Collaboration Boundary)

| 영역 | AI 담당 | 인간 담당 |
|------|---------|----------|
| 코드 생성 | 초안 작성, 패턴 적용 | 아키텍처 결정, 비즈니스 로직 판단 |
| 검증 | 자동 테스트, 갭 탐지, 교차 검증 | 최종 승인, 엣지 케이스 판단 |
| 규칙 관리 | 패턴 감지, 규칙 제안 | 승인/거부, 방향성 결정 |
| 문서화 | 초안 생성, 동기화 유지 | 의도/맥락 기술, 핵심 결정 기록 |
| 보안 | 취약점 스캔, 패턴 기반 탐지 | 위협 모델링, 보안 정책 수립 |

**원칙: AI는 제안하고, 인간은 결정한다.**

## 5. 컨텍스트 체인 (Zero Context Loss)

세션이 끊겨도, 팀원이 바뀌어도, 맥락은 살아있다.

### 3계층 메모리 아키텍처

```
Layer 1: Ephemeral (휘발성)
  └── 현재 대화 컨텍스트, 작업 중 상태
  └── 수명: 세션 내

Layer 2: Persistent (영속성)
  └── 프로젝트 메모리, 피드백, 의사결정 기록
  └── 수명: 프로젝트 생애주기
  └── 저장소: .claude/memory/, CLAUDE.md

Layer 3: Structural (구조적)
  └── 설계 문서, git 히스토리, 린터 규칙
  └── 수명: 영구
  └── 저장소: docs/, git log, .eslintrc
```

### 컨텍스트 복원 프로토콜

새 세션 시작 시:
1. CLAUDE.md 로드 → 프로젝트 전체 맥락 파악
2. 메모리 인덱스(MEMORY.md) 확인 → 최근 의사결정/피드백 복원
3. git log 확인 → 최근 작업 흐름 파악
4. 진행 중 태스크 확인 → 중단된 작업 이어서 진행

## 6. PDCA 통합 (실행 프레임워크)

AXIS는 PDCA 사이클 위에서 동작한다.

```
Plan    → CPS로 문제 정의, MECE로 요구사항 분해, 설계 문서 작성
Do      → 하네스 규칙 아래에서 구현, 린터 강제
Check   → X-Verification (교차 검증 + 역방향 검증 + 갭 분석)
Act     → Adaptive 사이클로 규칙 갱신, 개선 반영
```

### 품질 게이트

| 단계 | 통과 기준 | 실패 시 |
|------|----------|---------|
| Plan → Do | 설계 문서 완성, 린터 규칙 확인 | Do 진입 불가 |
| Do → Check | 빌드 성공, 기본 테스트 통과 | Check 진입 불가 |
| Check 갭 분석 | 설계-구현 일치율 90% 이상 | Act에서 자동 개선 반복 (최대 5회) |
| Act → 완료 | 교차 검증 합의율 90% 이상 | 사람 개입 후 재검토 |

## 7. 보안 체계 (Security by Design)

### AI 고유 위험 대응

| 위험 | 대응 |
|------|------|
| Prompt Injection | 입출력 검증 레이어, 구조화된 프롬프트만 사용 |
| 모델 환각(Hallucination) | 교차 검증으로 단일 모델 맹신 방지 |
| API 키 노출 | .env/.secret gitignore, 암호화 백업, 키 로테이션 |
| 자동 생성 코드 취약점 | OWASP Top 10 자동 스캔, 사람 보안 리뷰 |

### 규칙 변경 보안

- 규칙 변경은 반드시 사람 승인 필요 (AI 독단 변경 불가)
- 변경 이력 git 추적, 감사 로그 유지
- 보안 관련 규칙 변경은 2인 이상 승인 필요

## 9. 성공 지표 (KPIs)

| 지표 | 측정 방법 | 목표 |
|------|----------|------|
| 멱등성 점수 | 같은 요구사항 → 다른 AI → 결과 유사도 | 85% 이상 |
| 갭 탐지율 | 역방향 검증으로 발견한 설계-구현 불일치 | 자동 탐지 90% |
| 컨텍스트 복원율 | 새 세션에서 이전 맥락 복원 성공률 | 95% 이상 |
| 규칙 진화율 | 프로젝트 당 자동 제안된 규칙 채택률 | 60% 이상 |
| 교차 검증 효과 | 교차 검증으로 사전 발견한 결함 비율 | 단일 AI 대비 30% 향상 |

## 10. 하네스 엔지니어링과의 관계

```
하네스 엔지니어링 (Foundation)
  ├── 멱등성, CPS, MECE, 린터, Eval
  └── 정적 규칙 체계

AXIS Engineering (Evolution)
  ├── 하네스의 모든 것을 계승
  ├── + Adaptive (규칙이 살아있다)
  ├── + X-Verification (혼자 믿지 않는다)
  ├── + Context Chain (잊지 않는다)
  └── + Human-AI Boundary (역할을 나눈다)
```

AXIS는 하네스 엔지니어링의 대체가 아닌 **확장**이다.
