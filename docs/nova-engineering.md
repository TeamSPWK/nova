# Nova Engineering v1.0

> AI 시대의 소프트웨어 개발 방법론
> **N**ew Standards · **O**rchestrated · **V**erified · **A**daptive
>
> N — New Standards  : AI 개발의 새로운 기준을 세운다
> O — Orchestrated   : 멀티 에이전트를 체계적으로 조율한다
> V — Verified       : 독립 검증으로 품질을 보장한다
> A — Adaptive       : 규칙이 프로젝트와 함께 진화한다

---

## 1. 철학

**"처음부터 제대로 만들어 재작업을 제거하고, 매번 더 빠르게 만든다. 일관성이 속도를 만들고, 좋은 혁신은 즉시 흡수한다."**

기존 하네스 엔지니어링의 일관성 철학을 계승하되, 정적인 규칙 체계를 **살아있는 시스템**으로 진화시킨다. AI는 도구가 아니라 팀원이며, 인간과 AI의 역할 경계를 명확히 하여 각자의 강점을 극대화한다. 궁극적 목표는 **생산성** — 같은 시간에 더 높은 품질의 결과물을 더 많이 만드는 것이다.

---

## 2. 생산성의 진짜 병목

AI 도구는 코드 생성 속도를 극적으로 높였다. 그러나 **빠르게 만드는 것**과 **빠르게 완성하는 것**은 다르다. 진짜 병목은 코드 작성이 아니라 그 전후에 있다.

| 병목 | 증상 | Nova 해결책 |
|------|------|-------------|
| **방향 오류** | 잘못된 전제로 빠르게 구현 → 전면 재작업 | **Structured** — CPS로 문제를 먼저 정의, MECE로 빠짐없이 분해 |
| **AI 환각 누적** | 한 모델의 오류가 다음 작업으로 전파 → 디버깅 지옥 | **X-Verification** — 다관점 수집으로 환각을 조기 차단 |
| **암묵지 소실** | 세션/팀원 교체 시 맥락 유실 → 동일 실수 반복 | **Adaptive + Context Chain** — 규칙으로 암묵지를 명시화, 메모리로 맥락 유지 |
| **설계-구현 괴리** | 설계와 코드가 따로 놀아 → 품질 저하, 유지보수 불가 | **Gap Analysis** — 역방향 검증으로 괴리를 자동 탐지 |

Nova의 모든 구조는 이 네 가지 병목을 제거하기 위해 존재한다.

## 3. 핵심 원칙 (4 Pillars)

### Pillar 1: Structured — 구조가 품질을 만든다

> **생산성 기여**: 방향 오류를 원천 차단하여 재작업을 제거한다.

하네스 엔지니어링의 기반을 그대로 채택한다.

| 요소 | 설명 |
|------|------|
| CPS 프레임워크 | Context → Problem → Solution 순서로 사고 구조화 |
| MECE 분석 | 요구사항을 겹침 없이, 빠짐 없이 분해 |
| 문서 산출물 | 역기획 포함, 설계→구현 추적 가능한 구조화된 문서 |
| 린터 강제화 | 컨벤션을 사람의 의지가 아닌 시스템이 강제 |
| Eval 지표 | 조직 맞춤 품질 기준으로 측정 가능한 결과물 |

### Pillar 2: Consistent — 누가 해도 같은 프로세스와 품질 기준선

> **생산성 기여**: 일관성이 속도를 만든다 — 예측 가능한 프로세스는 리뷰와 통합 시간을 단축한다.

```
입력(요구사항) + 하네스(규칙 체계) = 일관된 프로세스와 품질 기준선
```

LLM 출력은 본질적으로 비결정적이므로, 같은 프롬프트에 같은 결과를 보장하지 않는다. Nova가 보장하는 것은 **결과의 동일성이 아니라 프로세스의 일관성**이다.

- **모델 독립성**: Claude, GPT, Gemini 어떤 AI를 사용해도 같은 프로세스와 품질 기준선
- **사람 독립성**: 시니어든 주니어든 하네스를 따르면 기준선 보장
- **시간 독립성**: 6개월 뒤 돌아와도 읽고 유지보수 가능
- **운영 멱등성**: 스크립트 재실행(init, install 등)은 안전 — 중복 생성이나 덮어쓰기 없음

### Pillar 3: X-Verification — 다관점 수집으로 판단력 향상

> **생산성 기여**: AI 환각과 설계-구현 괴리를 조기에 잡아 후반부 디버깅 비용을 제거한다.

단일 AI의 판단을 맹신하지 않는다. 여러 관점을 수집하여 판단 재료를 풍부하게 한다.

**적용 범위:** 기술 스택 선택, 아키텍처 판단 등 "정답이 없는 설계 결정"에 효과적이다. 사실 확인(factual question)에는 적합하지 않다 — 대부분의 AI가 같은 답을 내므로 합의가 높아도 의미 없다.

**3단계 검증 체계:**

```
Level 1: Self-Review     — 생성한 AI가 스스로 검토
Level 2: Multi-Perspective — 다른 AI/관점에서 다관점 수집
Level 3: Human-Review    — 사람이 최종 판단
```

**검증 기준 (Evaluator가 확인할 항목):**

| 항목 | 확인 내용 |
|------|----------|
| **기능** | 요청한 것이 실제로 동작하는가? CLAUDE.md 또는 NOVA-STATE.md의 요구사항 원문과 대조 |
| **데이터 관통** | 입력 → 저장 → 로드 → 표시까지 완전한가? 트리거 이후 사용자에게 실제 전달되는가? |
| **설계 정합성** | 기존 코드/아키텍처와 일관되는가? |
| **크래프트** | 에러 핸들링, 엣지 케이스, 타입 안전성 |
| **경계값** | 도메인 핵심 로직의 0, 음수, 빈 문자열, 최대값 등 경계 입력에서 크래시 없이 동작하는가? |

> "코드가 존재한다" ≠ "동작한다". 테스트 통과 ≠ 검증 완료. 가능한 경우 실제 실행 테스트를 수행한다.
> 도메인 핵심 로직(특히 금융/계산/인증)은 경계값 크래시 여부를 추가 확인한다.

**배포 후 검증 체크리스트** (배포가 포함된 작업 시 필수):
- 환경변수: 컨테이너/서버에 실제 반영되었는지 확인 (`docker exec`, `printenv` 등)
- 엔드포인트: curl로 주요 API 응답 확인
- 로그: 에러 로그 없는지 확인
- 사용자 흐름: 핵심 경로 1개 이상 수동 확인

**환경 변경 3단계**: 현재값 확인 → 변경 적용 → 반영 확인. 3단계를 모두 거치지 않으면 완료로 보지 않는다.

**합의 프로토콜:**

> ⚠️ 합의 수준은 AI의 정성적 판단이며, 정량 메트릭이 아니다. 참고 지표로만 사용한다.

- **Strong Consensus** (높은 합의) → 관점 간 핵심 방향 일치. 사람이 확인 후 진행.
- **Partial Consensus** (부분 합의) → 공통점과 차이점 공존. AI가 차이점 요약 후 사람이 판단.
- **Divergent** (관점 분산) → 관점 간 방향 불일치. 사람 필수 개입, 요구사항 재정의 검토.

**알려진 한계:**

- **공유 학습 편향**: Claude, GPT, Gemini는 학습 데이터의 상당 부분을 공유한다. 3개 모델이 합의했더라도 동일한 편향을 공유하는 것일 수 있다. 특히 사실 확인(factual question)이나 학습 데이터에 편향이 있는 주제에서는 Strong Consensus가 정확성을 보장하지 않는다.
- **"모두 틀림" 가드레일**: 3개 AI가 동의하는데 전부 틀린 경우를 방지하기 위해, 합의 결과에 항상 "이 주제에서 공유 편향 가능성이 있는가?" 체크포인트를 포함한다. 니치 프레임워크, 내부 시스템, 최신 API 등 학습 데이터에 충분하지 않은 영역에서는 Strong Consensus라도 인간 전문가 검증을 권장한다.
- **다관점의 한계**: "다른 AI"가 반드시 "다른 관점"은 아니다. 진정한 다관점을 원한다면 도메인 전문가, 사용자, 운영팀 등 인간의 관점을 추가하는 것이 가장 효과적이다.

**Generator-Evaluator 분리 원칙:**
- 구현하는 AI(Generator)와 검증하는 AI(Evaluator)는 독립적으로 동작
- 자기 평가 편향을 제거: "모델은 자기 결과물을 칭찬하는 경향이 있다"
- 설계 문서의 "검증 계약"이 Generator와 Evaluator 사이의 합의 기준
- 검증(Evaluator)은 **반드시** 독립 서브에이전트로 실행한다. 예외 없음.
- 검증 에이전트는 적대적 자세: "통과시키지 마라, 문제를 찾아라."

**역방향 검증 (Bidirectional Traceability):**
- 정방향: 설계 → 구현 (의도대로 만들었는가?)
- 역방향: 구현 → 설계 (구현이 설계를 벗어나지 않았는가?)
- 갭 탐지 시 자동 리포트 생성 → PDCA 루프 진입
- 개선 추세면 계속, 정체되면 접근 전환 (Iterative Refinement vs Pivot)

### Pillar 4: Adaptive — 규칙이 스스로 진화한다

> **생산성 기여**: 좋은 혁신을 즉시 흡수하여 매번 더 빠르게 만든다 — 암묵지를 규칙으로 전환해 반복 실수를 없앤다.

하네스는 고정이 아니다. 프로젝트와 함께 성장한다.

**진화 사이클:**

```
감지(Detect) → 제안(Propose) → 승인(Approve) → 적용(Apply) → 검증(Verify)
```

1. **감지**: AI가 반복 패턴 3회 이상 탐지
2. **제안**: "이 패턴을 컨벤션으로 등록할까요?" 자동 제안
3. **승인**: 사람이 검토 후 승인/거부/수정 (AI가 독단적으로 규칙 변경 불가)
4. **적용**: 승인된 규칙이 린터/CLAUDE.md/설계 문서에 자동 반영
5. **검증**: 새 규칙 적용 후 기존 코드와의 충돌 여부 자동 확인

**감지 메커니즘의 한계와 실제 동작:**

"3회 반복 감지"는 AI가 자동으로 세션 간 패턴을 추적한다는 의미가 아니다. Claude Code의 컨텍스트 윈도우는 단일 세션에 한정되므로, 감지는 다음 두 가지 방식으로 동작한다:

- **세션 내 감지**: 한 세션 안에서 같은 패턴이 반복될 때 AI가 자동 제안. 이것은 즉시 작동한다.
- **세션 간 감지**: CLAUDE.md와 `rules-changelog.md`에 기록된 이력을 기반으로 AI가 패턴을 인식. 이것은 AI가 CLAUDE.md를 읽고 "이전에도 비슷한 패턴이 있었다"고 판단할 때 작동한다.

즉, 세션 간 감지는 **파일에 기록된 이력에 의존하며**, AI의 자동 카운팅이 아니라 CLAUDE.md + git 히스토리 기반의 휴리스틱이다. 이 방식의 정확도는 기록의 충실도에 비례한다.

**최소 설계 기록:**

Plan/Design을 생략하는 소규모 프로젝트도 비기능 요구사항(인증 방식, 외부 연동 인터페이스, 배포 환경 전환 조건)은 CLAUDE.md에 기록한다. 기록이 없으면 갭 분석 시 "의도적 미구현"과 "누락"을 구분할 수 없다.

**변경 관리:**
- 모든 규칙 변경은 버전 관리 (git 추적)
- 롤백 가능: 규칙 변경이 품질 저하를 유발하면 즉시 되돌리기
- 변경 이력 대시보드로 "왜 이 규칙이 생겼는지" 추적 가능

## 3.5. Adaptive Verification (적응형 검증)

> 사용자에게 검증 강도를 매번 물어보는 것은 피로도를 높인다. 대신, AI가 변경 사항의 위험도를 자체 평가하여 검증 강도를 자동 스케일링한다.

### Risk Assessment 로직

작업 지시(Prompt)와 코드 변경량(git diff)을 기반으로 위험도를 1단계로 평가한다:

| Risk | 변경 영역 | 파일 수 | 검증 수준 |
|------|----------|---------|----------|
| **Low** | README, 설정, 스타일, 오타 | 1~2 | **Lite** — Senior Dev 단일 리뷰 |
| **Medium** | 새 컴포넌트, 내부 로직 수정 | 3~7 | **Standard** — Evaluator 3단계 (정적+의미론+실행) |
| **High** | DB 스키마, 결제, 인증, 아키텍처 | 8+ | **Full** — 다관점 자문 + Mutation Test + 적대적 Evaluator |

### 사용자 통제권 (Override)

기본은 AI 자동 판단(Auto)이지만, 사용자가 원할 때 강제할 수 있다:
- `--fast` : 무조건 Lite (토큰/시간 절약 우선)
- `--strict` : 무조건 Full (안전 우선)

### 비용/속도 효과

| 모드 | 토큰 사용량 (상대) | 응답 시간 (상대) |
|------|-------------------|-----------------|
| Lite | 1x (기준) | 1x |
| Standard | 3~5x | 2~3x |
| Full | 8~15x | 5~10x |

Lite 모드에서도 Generator-Evaluator 분리 원칙은 유지된다 (단일 리뷰어가 적대적 자세를 취함). Full 모드에서만 다중 에이전트와 교차검증이 추가된다.

## 3.6. 블로커 분류 (Blocker Classification)

구현 중 장애물을 만나면 다음 기준으로 분류한다:

| 분류 | 조건 | 대응 |
|------|------|------|
| **Auto-Resolve** | 외부 상태 변경 없이 되돌리기 가능 | 자동 해결 |
| **Soft-Block** | 진행 가능하나 런타임 실패 가능성 | 기록 후 계속 |
| **Hard-Block** | 데이터 손실/보안/돌이킬 수 없는 변경 | 즉시 중단, 사용자 판단 요청 |

불확실하면 Hard-Block으로 상향한다.

**자동 트리거**: 같은 원인의 실패가 2회 반복되면 블로커 분류를 강제한다. 분류 없이 3회 이상 수정→실패를 반복하지 않는다.

**코드 리뷰 시 블로커 기준** (구현 중 블로커와 별도):

| 기준 | 분류 |
|------|------|
| 런타임 크래시 유발: 특정 입력에서 예외/크래시 | Hard-Block |
| 데이터 손상/무결성 위반: FK 미검증, 잘못된 집계 | Hard-Block |
| 사용자 오판단 유발: 잘못된 금액/상태 표시 | Hard-Block |
| 기능 미동작: 정의만 있고 호출 안 됨, dead code | Soft-Block |

## 3.7. 세션 상태 유지 (NOVA-STATE)

- 프로젝트 루트에 `NOVA-STATE.md`가 있으면 세션 시작 시 반드시 읽는다.
- 상태 파일은 50줄 이내를 유지한다 — 인덱스 역할만, 상세는 링크로.
- 스프린트 전환 시 NOVA-STATE.md에 이력을 기록한다: "Sprint N 완료 → 확인 → Sprint N+1 시작".

**Known Gaps 필수**: 검증 후 발견된 미커버 영역을 NOVA-STATE.md에 기록한다. "ALL PASS"만 기록하면 과신을 유도한다. 미커버 경계값, 미테스트 경로, 알려진 제약을 명시한다.

**즉시 업데이트 트리거** (마지막에 몰아서 하지 않는다):
- 배포 성공/실패
- 테스트 통과/실패
- 스프린트 완료
- 블로커 발생/해소
- 검증(Evaluator) 결과 수신

## 4. 인간-AI 역할 경계 (Collaboration Boundary)

| 영역 | AI 담당 | 인간 담당 |
|------|---------|----------|
| 코드 생성 | 초안 작성, 패턴 적용 | 아키텍처 결정, 비즈니스 로직 판단 |
| 검증 | 자동 테스트, 갭 탐지, 다관점 수집 | 최종 승인, 엣지 케이스 판단 |
| 규칙 관리 | 패턴 감지, 규칙 제안 | 승인/거부, 방향성 결정 |
| 문서화 | 초안 생성, 동기화 유지 | 의도/맥락 기술, 핵심 결정 기록 |
| 보안 | 취약점 스캔, 패턴 기반 탐지 | 위협 모델링, 보안 정책 수립 |

**원칙: AI는 제안하고, 인간은 결정한다.**

## 5. 컨텍스트 체인 (Zero Context Loss)

세션이 끊겨도, 팀원이 바뀌어도, 맥락은 살아있다.

### 3계층 메모리 아키텍처

```
Layer 1: Ephemeral (휘발성)
  └── 현재 대화 컨텍스트, 작업 중 상태
  └── 수명: 세션 내

Layer 2: Persistent (영속성)
  └── 프로젝트 메모리, 피드백, 의사결정 기록
  └── 수명: 프로젝트 생애주기
  └── 저장소: .claude/memory/, CLAUDE.md

Layer 3: Structural (구조적)
  └── 설계 문서, git 히스토리, 린터 규칙
  └── 수명: 영구
  └── 저장소: docs/, git log, .eslintrc
```

### 컨텍스트 복원 프로토콜

새 세션 시작 시:
1. CLAUDE.md 로드 → 프로젝트 전체 맥락 파악
2. 메모리 인덱스(MEMORY.md) 확인 → 최근 의사결정/피드백 복원
3. git log 확인 → 최근 작업 흐름 파악
4. 진행 중 태스크 확인 → 중단된 작업 이어서 진행

## 6. PDCA 통합 (실행 프레임워크)

Nova는 PDCA 사이클 위에서 동작한다.

```
Plan    → CPS로 문제 정의, MECE로 요구사항 분해, 설계 문서 작성
Do      → 하네스 규칙 아래에서 구현, 린터 강제
Check   → X-Verification (다관점 수집 + 역방향 검증 + 갭 분석)
Act     → Adaptive 사이클로 규칙 갱신, 개선 반영
```

### 품질 게이트

| 단계 | 통과 기준 | 실패 시 |
|------|----------|---------|
| Plan → Do | 설계 문서 완성, 린터 규칙 확인 | Do 진입 불가 |
| Do → Check | 빌드 성공, 기본 테스트 통과 | Check 진입 불가 |
| Check 갭 분석 | 설계-구현 일치율 90% 이상 | Act에서 자동 개선 반복 (최대 5회) |
| Act → 완료 | 다관점 수집에서 Strong Consensus | 사람 개입 후 재검토 |

## 7. 보안 체계 (Security by Design)

### AI 고유 위험 대응

| 위험 | 대응 |
|------|------|
| Prompt Injection | 입출력 검증 레이어, 구조화된 프롬프트만 사용 |
| 모델 환각(Hallucination) | 다관점 수집으로 단일 모델 맹신 방지 |
| API 키 노출 | .env/.secret gitignore, 암호화 백업, 키 로테이션 |
| 자동 생성 코드 취약점 | OWASP Top 10 자동 스캔, 사람 보안 리뷰 |

### 규칙 변경 보안

- 규칙 변경은 반드시 사람 승인 필요 (AI 독단 변경 불가)
- 변경 이력 git 추적, 감사 로그 유지
- 보안 관련 규칙 변경은 2인 이상 승인 필요

## 9. 성공 지표 (KPIs)

> ⚠️ 아래 수치는 **도입 목표**이며, 실측 결과가 아니다. Nova는 아직 이 지표들에 대한 통계적으로 유의미한 before/after 데이터를 보유하지 않는다. 실제 프로젝트에서의 측정 결과가 축적되면 이 섹션을 업데이트할 예정이다.

| 지표 | 측정 방법 | 목표 |
|------|----------|------|
| 프로세스 일관성 | 같은 요구사항 → 다른 AI → 프로세스·구조 준수율 | 85% 이상 |
| 갭 탐지율 | 역방향 검증으로 발견한 설계-구현 불일치 | 자동 탐지 90% |
| 컨텍스트 복원율 | 새 세션에서 이전 맥락 복원 성공률 | 95% 이상 |
| 규칙 진화율 | 프로젝트 당 자동 제안된 규칙 채택률 | 60% 이상 |
| 다관점 수집 효과 | 다관점 수집으로 사전 발견한 결함 비율 | 단일 AI 대비 30% 향상 |

## 10. 하네스 엔지니어링과의 관계

```
하네스 엔지니어링 (Foundation)
  ├── 일관성, CPS, MECE, 린터, Eval
  └── 정적 규칙 체계

Nova Engineering (Evolution)
  ├── 하네스의 모든 것을 계승
  ├── + Adaptive (규칙이 살아있다)
  ├── + X-Verification (혼자 판단하지 않는다)
  ├── + Context Chain (잊지 않는다)
  └── + Human-AI Boundary (역할을 나눈다)
```

Nova는 하네스 엔지니어링의 대체가 아닌 **확장**이다.
