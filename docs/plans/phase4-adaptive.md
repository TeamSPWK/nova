## Context (배경)

### 현재 상태
- Phase 1~3 완성: CPS 구조, 교차검증, 갭 체크, 컨텍스트 체인
- 규칙(컨벤션)은 CLAUDE.md에 정적으로 기록되어 있음
- 프로젝트가 성장하면 새로운 패턴이 등장하지만, 이를 체계적으로 포착하는 구조가 없음

### 왜 필요한가
- AXIS의 핵심 차별점: "규칙이 프로젝트와 함께 진화한다"
- 하네스 엔지니어링은 정적 규칙 → AXIS는 살아있는 규칙
- 규칙 변경의 이력 관리 없이는 "왜 이 규칙이 있는지" 추적 불가

### 관련 자료
- `docs/axis-engineering.md` Pillar 4: Adaptive
- 진화 사이클: 감지 → 제안 → 승인 → 적용 → 검증

## Problem (문제 정의)

### 핵심 문제
프로젝트의 반복 패턴을 규칙으로 승격시키는 **체계적 프로세스**가 없다.

### MECE 분해

| # | 문제 영역 | 설명 | 영향도 |
|---|----------|------|--------|
| 1 | 규칙 제안 체계 부재 | AI가 패턴을 발견해도 제안할 구조화된 방법이 없음 | 높음 |
| 2 | 변경 이력 추적 없음 | 규칙이 언제, 왜 바뀌었는지 모름 | 높음 |
| 3 | 승인 프로세스 없음 | AI가 독단적으로 규칙 변경할 위험 | 중간 |

### 제약 조건
- 경량 도구 키트 — 자동 감지 시스템(코드 분석기)은 범위 밖
- 사람이 트리거하는 구조 (AI가 제안, 사람이 승인)
- 파일 기반, git 추적 가능

## Solution (해결 방안)

### 선택한 방안
**규칙 제안 커맨드 + 규칙 변경 로그** — `/propose` 커맨드로 규칙 제안, 변경 이력은 CHANGELOG 형식

### 구현 범위
- [ ] `/propose` 슬래시 커맨드 — 규칙 제안 문서 생성
- [ ] `docs/templates/rule-proposal.md` — 규칙 제안 템플릿
- [ ] `docs/rules-changelog.md` — 규칙 변경 이력
- [ ] `docs/axis-engineering.md` Phase 4 체크리스트 업데이트

### 검증 기준
- `/propose` 커맨드로 구조화된 규칙 제안서 생성 가능
- 규칙 변경 이력이 시간순으로 추적 가능
- 기존 5개 커맨드와 일관된 구조
