## Context (배경)

### 현재 상태

Nova는 5기둥(환경·맥락·품질·협업·진화) AI Agent Ops 프레임워크이고, "품질" 기둥에 Generator–Evaluator 분리 + Hard Gate라는 실행 계약이 있다. `docs/nova-rules.md` §2는 "Evaluator PASS 없이 커밋 시 `hooks/pre-commit-reminder.sh`가 exit 2로 차단한다"고 명시한다. Always-On 규칙 #3(독립 Evaluator)과 #4(커밋 전 /nova:review --fast)도 이 Hard Gate에 의해 강제되도록 설계됐다.

동시에 v5.13.0 Sprint 1에서 `.nova/events.jsonl` + Privacy Filter 14 정규식 + flock/mkdir fallback + rotation을 구축했고, v5.17.0 Tier 3에서 `analyze-observations.sh`가 이 관측 데이터를 학습 재료로 쓴다. v5.18.0~v5.18.2는 `pre-tool-use-record.sh`의 PreToolUse 훅을 stdin JSON 파싱으로 전환해 Bash 이벤트 기록을 실측으로 확인했다.

### 왜 필요한가

2026-04-23 Nova 팀 자가 감사 결과 3층 붕괴가 드러났다.

| 층위 | 실측 증거 | 영향 |
|------|----------|------|
| L1 Enforcement | `hooks/pre-commit-reminder.sh:7`이 `$TOOL_INPUT` env var 가정 (v5.18.2가 확정한 stdin JSON 스펙과 불일치). Nova 팀 당일 4회 릴리스 동안 commit 차단 0회 | **현재 배포된 플러그인 사용자 전체의 Evaluator Hard Gate가 no-op 상태** |
| L2 Observability | `.nova/events.jsonl` event_type 분포: {733 session_start, 89 tool_call, 57 session_end, 10 test_event, 4 orchestration_missing, 1 jury_verdict}. 규칙 준수 이벤트 0건 | Tier 3 Behavior Learning이 자기 규칙 준수를 학습·진단할 데이터 없음 |
| L3 Pipeline | `scripts/release.sh` Step 3의 `git commit`이 pre-commit-reminder 차단 결과에 의존하지만 훅이 고장나도 감지 불가 | release 파이프라인이 "hook 생존"을 전제로만 작동 |

기존 메모 `project_nova_rule_self_enforcement_gap_2026_04_23.md`는 "release.sh에 review/evaluator/state 게이트 통합"이라 프레임했지만, 이는 **증상 진단**이었다. 실제 근본 원인은 훅 자체가 고장 났고(L1), 관측이 없어(L2) 이를 감지하지 못했으며, 파이프라인이 훅 생존을 맹신했다(L3)는 점이다.

"규칙 자발 미준수"로 보였던 현상은 **훅 집행 구조 붕괴**였다. 이것은 Nova 팀만의 문제가 아니라 플러그인 배포를 받은 모든 사용자에게 적용되는 구조적 결함이다.

## Problem (문제 정의)

### 핵심 문제

**Nova가 문서로 선언한 품질 규칙(§2 Hard Gate, §10 관측 계약, Always-On #3/#4)이 실제로 실행되지 않는다.** 배포된 플러그인은 "차단되는 것처럼 보이지만 실제로는 차단하지 않는" 상태다.

### MECE 분해

|---|----------|------|--------|

### 제약 조건

**구조적 제약 (code-explorer 조사)**

1. stdin 바인딩 형태는 v5.18.2의 `pre-tool-use-record.sh` 패턴을 재사용해야 한다 (`IFS= read -r INPUT_JSON`, `jq -r '.tool_input.command' <<<"$INPUT_JSON"`).
2. NOVA-STATE.md Hard Gate 상태 머신은 기존 6상태(PASS/MISSING/EMPTY/NO_PASS/STALE/TIMESTAMP_BROKEN) + 신규 1상태(CONFLICT — `<<<<<<<` merge 마커 감지)로 **7가지**로 확장한다. CONFLICT는 fail-closed(exit 2).
3. `.nova/events.jsonl`는 schema_version v1을 유지하며 신규 이벤트도 필수 필드(timestamp, session_id, event_type, nova_version, redacted 등)를 따른다. 신규 이벤트는 **`files: [{path, sha256}]`** 필드를 선택적으로 포함할 수 있다(review_pass/review_fail/commit_blocked/commit_emergency에서 필수, S3.2 ledger 매칭 키로 사용).
4. hooks.json 변경은 Claude Code 완전 재시작 시점에만 반영된다 (`/reload-plugins`로 부족 — memory reference_claude_code_hooks_mechanics.md). 훅을 건드린 Sprint의 릴리스 전 실기 테스트는 **반드시 완전 재시작 이후** 수행한다.
5. session-start.sh soft 1900B / hard 2500B 예산을 초과하지 않는다 (이 Plan에서 새 규칙 섹션을 추가하면 session-start.sh는 §번호 참조만 유지).

**운영 제약**

6. Nova는 범용 플러그인 — 예제 fixture에 특정 프로젝트명을 박지 않는다 (memory: feedback_nova_universal_plugin.md).
7. Release workflow: `scripts/release.sh` 경로 유지. `bump-version.sh` + landing 자동 동기화 계약 깨지 않는다.
8. 회귀 방지 assert(v5.18.1 S8.1b matcher regex, v5.13.0 S 시리즈)는 모두 유지·확장한다.
9. **레거시 스크립트 forward-compat** — v5.18.x 이하의 analyze-observations, nova-metrics, 외부 툴이 신규 event_type이 섞인 jsonl을 파싱할 때 **unknown event_type은 skip (not fail)** 이어야 한다. S2.7의 backward-compat 테스트로 보증.
10. **네이티브 git 훅 레이어 경계** — Claude Code 훅(PreToolUse)은 **Claude가 Bash 도구로 git 명령을 실행하기 전 시점**에 동작한다. 네이티브 git 훅(husky/lefthook/.git/hooks/\*)은 **실제 git 바이너리 실행 시점**에 동작한다. 두 레이어는 독립이며 Nova는 Claude Code 훅 레이어만 담당한다. 네이티브 훅이 별도로 설치돼 있어도 간섭하지 않으며, 사용자가 터미널에서 직접 git commit을 친 경우 Nova Hard Gate는 작동하지 않는다(설계 의도).

## Solution (해결 방안)

### 선택한 방안

**Option A — 분리 릴리스**: `v5.18.3 (S1 hotfix: patch)` → 2주 내 `v5.19.0 (S2+S3+S4: minor)`.

option-explorer 분석에 근거한 선택 근거:

- **사용자 신뢰 복구 속도**: 현재 배포된 플러그인의 Hard Gate가 no-op이다. v5.18.3 hotfix를 2~3일 내 릴리스하면 **모든 플러그인 사용자가 Hard Gate 복구 혜택을 즉시 받는다**. 단일 v5.19.0으로 묶으면 사용자는 S2~S4 개발 기간(2~3주) 내내 no-op 상태로 방치된다.
- **테스트 부담 분산**: v5.18.3은 S1 단일 변경에 집중하므로 실기 테스트가 명확(재시작 1회 + 케이스 3종). v5.19.0은 S1이 먼저 안정화된 기반 위에서 S2/S3/S4 조합을 검증한다.
- **n=1 개발자 피로**: 오늘 4회 릴리스 누적. 단일 v5.19.0 묶음(Option B)은 n=1 개발자에게 테스트 부담 폭증. Option C(점진 3+회)는 반복 릴리스 오버헤드로 오히려 피로 가중.
- **버전 범프 정합성**: S1은 버그 수정이므로 patch 수준이 맞다. S2~S4는 새 이벤트 타입·커맨드 기능 확장이므로 minor 수준.

**S1.1 pre-commit-reminder.sh 구체화 (U2 해소 반영, fail-closed)**

```bash
#!/usr/bin/env bash
# Nova Engineering — PreToolUse Hook (Bash) — v5.18.3
# stdin JSON 파싱 (v5.18.2 pattern + 공식 스펙)

# ── 우회 스위치 (테스트 편의)
if [ "${NOVA_DISABLE_EVENTS:-0}" = "1" ]; then
  exit 0
fi

# ── stdin 파싱 (fail-closed: tty/미입력/jq 오류 시 exit 2)
if [[ ! -t 0 ]]; then
  INPUT_JSON=$(cat 2>/dev/null || true)
  if [ -z "$INPUT_JSON" ]; then
    exit 2   # fail-closed: 빈 stdin
  fi
  COMMAND=$(echo "$INPUT_JSON" | jq -r '.tool_input.command // empty' 2>/dev/null || true)
  if [ $? -ne 0 ]; then
    exit 2   # fail-closed: jq 파싱 실패
  fi
else
  # tty 수동 실행 (사용자 디버그) — 차단 안 함
  exit 0
fi

# ── git commit 패턴 감지
if echo "$COMMAND" | grep -qE '^\s*git\s+(.*\s+)?commit(\s|$)'; then
  # (상태 머신 7가지 — PASS/MISSING/EMPTY/NO_PASS/STALE/TIMESTAMP_BROKEN/CONFLICT)
  # ... (본문은 기존 로직 + CONFLICT 분기 추가)
  :
fi
```

- **fail-closed 정책**: stdin 부재(tty 제외), jq 실패, readonly FS로 인한 `.nova/events.jsonl` append 실패 — 모두 exit 2.

### Pre-Release Checklist (훅 변경 포함 릴리스 필수)

> hooks.json 또는 `hooks/*.sh`가 변경된 릴리스에서는 아래 순서를 **반드시** 따른다. 순서가 뒤바뀌면 실기 테스트가 캐시된 훅으로 실행되어 거짓 PASS가 나올 수 있다.

```
┌─────────────────────────────────────────────────────────────────┐
│ PRE-RELEASE CHECKLIST (hooks 변경 릴리스 전용)                  │
├─────────────────────────────────────────────────────────────────┤
│ 1. [ ] hooks.json 또는 hooks/*.sh 변경 감지                     │
│       → git diff HEAD~1 -- hooks/ 가 non-empty 인지 확인         │
│                                                                  │
│ 2. [ ] Claude Code 완전 재시작 (`/reload-plugins` 로는 부족)    │
│       → 세션 종료 → `claude` 재실행                              │
│       → memory reference_claude_code_hooks_mechanics.md 참조     │
│                                                                  │
│ 3. [ ] 재시작 후 실기 테스트 실행                                │
│       → S1-V10 / S2-V10 / S3-V9 / S4-V7 해당 항목                │
│       → 각 항목은 "Claude Code 완전 재시작 후" 전제              │
│                                                                  │
│ 4. [ ] 실기 테스트 결과 NOVA-STATE.md Last Activity 반영         │
│                                                                  │
│ 5. [ ] tests/test-scripts.sh 전체 실행 PASS                      │
│                                                                  │
│ 6. [ ] /nova:review --fast PASS → 독립 Evaluator PASS 확인       │
│                                                                  │
│ 7. [ ] scripts/release.sh <patch|minor> "…" 실행                 │
└─────────────────────────────────────────────────────────────────┘
```

S2/S3/S4만 변경된 릴리스에서도 hooks/*.sh가 건드려졌다면(S2.4/S2.5) 위 체크리스트 전체 적용. hooks 미변경 릴리스는 2·3·4 단계 생략 가능.
