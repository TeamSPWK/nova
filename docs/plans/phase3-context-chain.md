- Phase 2 (Verification): ✅ 완성 — x-verify.sh (교차검증), gap-check.sh (역방향 검증)

## Problem (문제 정의)

### 핵심 문제
axis-kit을 복사해서 쓰는 프로젝트에서, **세션이 끊겨도 맥락을 복원**할 수 있는 표준 체계가 없다.

### MECE 분해

| # | 문제 영역 | 설명 | 영향도 |
|---|----------|------|--------|
| 1 | 복원 프로토콜 부재 | 새 세션 시작 시 무엇을 확인해야 하는지 가이드가 없음 | 높음 |
| 2 | CLAUDE.md 템플릿 부재 | 프로젝트별 CLAUDE.md 작성 기준이 없어 품질이 들쑥날쑥 | 높음 |
| 3 | 의사결정 기록 구조 없음 | 왜 그런 결정을 했는지 추적 불가 (git log만으로 부족) | 중간 |
| 4 | 규칙 변경 추적 없음 | Phase 4(Adaptive) 준비를 위한 변경 이력 구조 필요 | 중간 |

### 제약 조건
- axis-kit은 **경량 도구 키트** — 무거운 인프라(DB, 서버) 없이 파일 기반만
- Claude Code의 기존 메모리 시스템(.claude/memory/)과 충돌하지 않아야 함
- 복사 후 바로 쓸 수 있어야 함 (설정 최소화)

## Solution (해결 방안)

### 선택한 방안
파일 기반 컨텍스트 체계 — 템플릿 + 커맨드로 구조화

### 대안 비교

| 기준 | A: 파일 기반 템플릿 | B: 전용 스크립트 자동화 | C: 외부 도구 연동 |
|------|-------------------|----------------------|------------------|
| 경량성 | ✅ 파일만 | ⚠️ 스크립트 의존 | ❌ 외부 도구 필요 |
| 즉시 사용 | ✅ 복사하면 끝 | ⚠️ 설정 필요 | ❌ 설치 필요 |
| AI 도구 독립 | ✅ 어떤 AI든 읽을 수 있음 | ⚠️ Claude 특화 가능 | ❌ 특정 도구 종속 |
| 선택 | **채택** | 기각 (과도한 복잡성) | 기각 (경량 원칙 위배) |

### 구현 범위

1. **CLAUDE.md 템플릿** (`docs/templates/claude-md.md`)
   - 프로젝트별 CLAUDE.md 작성 가이드
   - 필수 섹션: 프로젝트 개요, AXIS 설정, 커맨드, 컨벤션, 보안 규칙

2. **의사결정 기록 템플릿** (`docs/templates/decision-record.md`)
   - ADR(Architecture Decision Record) 경량 버전
   - CPS 구조 적용: Context → Problem → Decision + 결과

3. **컨텍스트 복원 가이드** (`docs/context-chain.md`)
   - 새 세션 시작 시 복원 프로토콜 문서화
   - 3계층 메모리의 실용적 매핑: 어떤 정보를 어디에 저장하는지

4. **README 업데이트**
   - Context Chain 섹션 추가
   - 설치 가이드에 CLAUDE.md 설정 단계 추가

### 검증 기준
- [ ] CLAUDE.md 템플릿으로 실제 프로젝트에 적용 가능한 CLAUDE.md 생성 가능
- [ ] 의사결정 기록 템플릿이 CPS 구조를 따름
- [ ] 컨텍스트 복원 가이드가 3계층 모두 커버
- [ ] 기존 Phase 1~2 산출물과 일관성 유지
