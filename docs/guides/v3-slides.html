Nova v3

AI 에이전트의 work-item registry

"오늘 뭐 했는지 · 다음 뭐 할지"를 기계가 읽을 수 있게

v5.43.2+ 5분 가이드

발표자: Nova v3는 AI와 사람이 함께 일하는 프로젝트에서 진척 상황을 기계가 읽을 수 있게 정리한 시스템입니다. 5분이면 핵심 흐름을 익힐 수 있습니다.

🎯 한 줄 요약

사용자는 NOVA-STATE.md 한 파일만 보면 충분.

슬래시 커맨드 (/nova:plan, /nova:run 등)가 work-item을 자동 관리.

🧑 사람용 cursor:   NOVA-STATE.md          ← 매일 한 번 보는 곳
🤖 기계용 진실원:   .nova/work-items/      ← AI가 자동 관리
핵심 분리: 사람과 기계가 각각 다른 곳을 본다. 한쪽이 무너져도 다른 한쪽은 살아있음.

📉 왜 필요한가?

❌ Before (v1/v2)

  • NOVA-STATE.md가 wiki처럼 커짐
  • AI가 stale 정보로 같은 작업 반복 추천
  • 사람이 매번 손으로 정리
  • 진척 측정 불가 (DORA 지표 없음)
  • commit_sha 추적 X

✅ After (v3)

  • AI가 자동 work-item 발급·전이
  • 사람은 cursor만 봄
  • .nova/work-items/가 단일 진실원
  • drift 18 룰로 무결성 자동
  • DORA 표준 evidence (commit_sha)
v1/v2는 wiki화 + drift가 가장 큰 문제. v3는 분리 + 단방향 투영으로 해결.

⚡ 5분 시작 — 시나리오 A

새 프로젝트

cd ~/my-project
/nova:setup

자동으로:

  • .nova/{schema,work-items,README.md} 생성
  • .gitignore Nova marker 블록 추가
  • 빈 매니페스트 index.json (next_seq=1)

→ 끝. 이후 슬래시 커맨드 사용 시 AI 자동 work-item 발급

신규 프로젝트는 한 명령으로 모든 게 준비됨. 사용자는 외울 게 없음.

⚡ 5분 시작 — 시나리오 B

기존 v1/v2 STATE 마이그레이션

cd ~/existing-project
/nova:migrate-state    # target 옵션 없이!

자동 흐름:

  1. NOVA-STATE.md schema 자동 감지 (v1/v2 모두)
  2. dry-run 미리보기 (보존율 + WI 분포 보고)
  3. 사용자 4지선다 검수 (A=apply / B=중단 / C=가이드 / D=skip)
  4. A 선택 → --apply + drift-check 자동

v1 → v3 직행 — multi-hop 불필요. --target=v2는 legacy 호환용 (deprecated).

v1과 v2 모두 같은 명령으로 처리. 사용자가 입력 schema를 알 필요 없음.

⚡ 5분 시작 — 시나리오 C

평상시 사용

/nova:plan "검색 기능 추가"     # → WI-0042 (proposed)
/nova:run                       # → status=done + commit_sha
/nova:review                    # → review_required=true
/nova:check                     # → drift 18 룰 자동 검증

사용자는 평소 그대로 슬래시 사용. 내부에서 work-item 자동 관리.

NOVA-STATE.md marker 영역만 봐도 진척 파악 OK ✨

평상시는 v1/v2와 동일. 변경된 건 내부 자동화만.

🔄 5 상태 enum

status의미자동 트리거
proposed계획만 잡힘/nova:plan
active진행 중/nova:run 시작
blocked막힘 (reason 필수)/nova:run 도중 블로커
done완료 + commitEvaluator PASS
superseded폐기/nova:plan으로 대체
원자적 보장: done 전이 시 review_required=false + evidence.commit_sha 동시 set. 부분 갱신 불가능 (SIGINT 안전).
5값 enum은 절대 늘리지 않음. GPT 강조: 상태가 늘어나면 시스템이 복잡해진다.

👀 사용자가 매일 보는 NOVA-STATE.md

<!-- nova:registry-rendered:start -->
// 이 안쪽은 .nova/work-items/index.json을 자동 투영
// 사람이 손편집 X — 다음 렌더 시 덮어쓰기

**Active Tree** (registry: 3 work-items, 갱신: 2026-05-16):

- 🔄 WI-0042-add-search — high
- ⬜ WI-0043-fix-payment — medium · review_required ⚠️
- ⬜ WI-0044-doc-update — low

**Recent Activity** (last 7d):

- 2026-05-16: WI-0042 — proposed → active
- 2026-05-15: WI-0041 — proposed → done (commit abc1234)
<!-- nova:registry-rendered:end -->

marker 외 영역(Current Goal, Tasks, Known Risks)은 사람 손편집 영역으로 보존

marker 안쪽은 자동 갱신, 바깥은 사람 자유. 분리가 핵심.

🛡️ drift 18 룰 — 자동 무결성

/nova:check 실행 시 자동 18 룰 검증

✅ exit 0

PASS — 보고만

⚠️ exit 1

Warn 9 — 안내 + 후속

❌ exit 2

Hard 9 — check FAIL

분류대표 룰
Hard 9H6 done evidence 부재 · H8 부분 전이 잔류 · H9 blocked_reason 빈
Warn 9W5 git 미커밋 · W6 UUID fallback · W7 source_docs 빈
Hard는 즉시 차단, Warn은 안내만. 마이그레이션 직후 W5/W7 발화는 정상.

💬 FAQ (1/2)

Q1. 기존 NOVA-STATE.md 사라지나요?

아니요. NOVA-STATE.md.v2.bak 자동 백업. marker 외 영역도 그대로 보존.

Q2. team 공유는?

.nova/work-items/·schema/·README.md = git-tracked. clone 즉시 복원.
events.jsonl·local/ = git-ignored (사용자별).

Q3. 다른 AI (Cursor, Copilot) 호환?

JSON Schema(v3.0) 기반. 어떤 도구도 .nova/work-items/*.json 직접 읽기 가능.

백업·공유·호환성 — 세 가지가 가장 자주 묻는 질문.

💬 FAQ (2/2)

Q4. UI/대시보드는?

/nova:status — HTML dashboard 자동 생성. Phase·Sprint 진행률 + drift 알람.

Q5. 마이그레이션 실패 시 복구?

mv NOVA-STATE.md NOVA-STATE.md.v3.failed
mv NOVA-STATE.md.v2.bak NOVA-STATE.md
rm -rf .nova/work-items .nova/schema .nova/README.md

Q6. NOVA-STATE.md만 보면 정말 충분?

일상 운영(매일 작업 시작·진척 확인·다음 행동)은 YES. 깊은 정보는 슬래시 커맨드가 대신 봄.

"NOVA-STATE.md만 보면 됨"이 v3의 사용자 약속.
