하드 게이트: Generator ≠ Evaluator

구현 완료
→
tsc/lint
→
Evaluator 서브에이전트
→
PASS
→
커밋 허용

🛡️ 기술적 정의: "독립 서브에이전트"

반드시 별도 서브에이전트 spawn으로 컨텍스트 창이 분리된 상태. 메인 에이전트가 직접 코드를 재읽고 "문제 없다"고 판단하는 것은 독립 검증이 아님.
자가 점검: "나는 이 코드를 구현한 에이전트와 동일 컨텍스트 창에 있는가?"

🚨 차단

Evaluator PASS 없이 커밋 시 hooks/pre-commit-reminder.sh가 exit 2로 차단. 예외: --emergency (프로덕션 장애 긴급 수정).

"사용자가 'Evaluator 돌렸니?'라고 물어보는 상황은 프로세스 실패다."

기둥 ④ 협업 Collaboration

X-Verification — 3개 AI 다관점 자문

$ /nova:ask "Redis vs Postgres LISTEN/NOTIFY 결정"

  Claude → "Postgres LISTEN이 연결 상태 복잡성 적음..."
  GPT    → "Redis가 초당 10만 메시지 이상 유리..."
  Gemini → "워크로드 정의 먼저. 초당 1000↓이면..."

  → 합의 수준: Partial Consensus (70~89%)
  → AI가 차이점 요약 → 사람이 판단
합의조건행동
Strong Consensus관점 핵심 방향 일치사람 확인 후 진행
Partial Consensus공통점+차이점 공존AI가 차이점 요약 → 사람 판단
Divergent방향 불일치사람 필수 개입, 요구사항 재정의

⚠️ 공유 학습 편향: 3 AI가 합의해도 같은 편향일 수 있음. 사실 확인(factual)엔 부적합.

  4. 커밋 전 /nova:review --fast 자동 실행

  6. 블로커 발생 시 즉시 사용자에게 알림

"Nova는 배경에서 자동으로 품질을 관리한다."

/nova:auto "기능 X 추가해줘"    # 설계→구현→검증 전체 사이클

실전 예시 — 복잡한 작업

"인증 미들웨어 교체" (10+ 파일 · 보안 영역)

# 1. DeepPlan (Explorer×3 병렬 탐색)
/nova:auto --deep "인증 미들웨어 교체"

# 자동 실행:
#   └─ Phase A: Explorer×3 → 기존 코드/리스크/대안 3종 병렬 탐색
#   └─ Phase B: Synthesizer → Plan 초안 (Risk Map + Unknowns + V-Hook)
#   └─ Phase C: Critic (evaluator) → 적대적 검증
#   └─ Phase D: Refiner (필요 시) → 이슈 반영
#   └─ /nova:design → Sprint Contract 구체화
#   └─ Sprint 1 구현 → Evaluator → PASS → 커밋
#   └─ Sprint 2 구현 → Evaluator → CONDITIONAL → 수정 → PASS
#   └─ 최종 /nova:review --strict → 릴리스

인증/DB/결제 = 자동 한 단계 상향 (§1). 자가 완화 금지 — AI가 "이 정도면 괜찮다"고 스킵 못함.
