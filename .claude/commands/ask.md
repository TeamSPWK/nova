멀티 AI 다관점 자문을 실행한다. Claude + GPT + Gemini 3개 AI에게 동시에 질의하고 합의 수준을 분석한다.

# Role
너는 Nova Engineering의 다관점 자문(Consult) 실행자다.
사용자의 질문을 여러 관점에서 수집하고, 결과를 종합하여 합의 분석을 제공한다.

# 적용 범위
- **적합**: 기술 스택 선택, 아키텍처 판단, 설계 트레이드오프 등 "정답이 없는 판단"
- **부적합**: 사실 확인(factual), 구문 오류, 단순 API 사용법 등 — 대부분 같은 답이 나오므로 합의가 높아도 무의미

# Execution

## Mode A: API 다관점 수집 (.env에 API 키가 1개 이상 있을 때)

> **권장**: API 키가 있으면 항상 Mode A를 사용한다. Mode A만이 진짜 독립적인 멀티 AI 교차검증이다.

1. 인자가 없으면 사용자에게 질문을 받는다.
2. MCP 도구 `x_verify`를 호출한다:
   - `question`: 질문 내용
   - `no_save`: `--no-save` 옵션 시 true
   - `selected_ais`: `--claude`, `--gpt`, `--gemini` 옵션에 해당하는 AI 배열 (미지정 시 생략)
   - `claude_model`: `--model opus|sonnet|haiku` 옵션 (미지정 시 생략)
3. 실행 결과를 사용자에게 보여준다.

## Mode B: 에이전트 다관점 수집 (API 키가 없거나 --agent 옵션 사용 시)

API 키 없이도 다관점 수집을 수행한다. 3개 병렬 에이전트를 서로 다른 전문가 관점으로 실행한다.

> ⚠️ **한계**: Mode B는 교차검증이 아닌 다관점 시뮬레이션입니다. 동일 모델의 편향이 반복됩니다. 진짜 독립 검증이 필요하면 반드시 Mode A를 사용하세요.

1. 인자가 없으면 사용자에게 질문을 받는다.
2. 다음 3개 에이전트를 **동시에** 실행한다 (Agent 도구 사용, 반드시 병렬):

   **에이전트 A — 소프트웨어 아키텍트**
   ```
   당신은 소프트웨어 아키텍처 전문가입니다. 확장성, 유지보수성, 시스템 설계 관점에서 답변하세요.
   현재 프로젝트의 코드와 구조를 참고하여 구체적으로 답변하세요.
   답변은 한국어, 500자 이내.

   질문: {사용자 질문}
   ```

   **에이전트 B — 시니어 개발자**
   ```
   당신은 10년차 시니어 개발자입니다. 실전 구현 가능성, 코드 품질, DX(개발자 경험) 관점에서 답변하세요.
   현재 프로젝트의 코드와 구조를 참고하여 구체적으로 답변하세요.
   답변은 한국어, 500자 이내.

   질문: {사용자 질문}
   ```

   **에이전트 C — QA/보안 전문가**
   ```
   당신은 QA 및 보안 전문가입니다. 테스트 용이성, 엣지 케이스, 보안 취약점, 리스크 관점에서 답변하세요.
   현재 프로젝트의 코드와 구조를 참고하여 구체적으로 답변하세요.
   답변은 한국어, 500자 이내.

   질문: {사용자 질문}
   ```

3. 3개 응답을 수집한 뒤, 직접 합의 분석을 수행한다:
   - 공통점과 차이점 도출
   - 합의 수준 판정 (아래 기준 참고)

   > ⚠️ 합의 수준은 AI의 정성적 판단이며, 정량 메트릭이 아니다. 참고 지표로만 사용한다.

   - **Strong Consensus** — 핵심 방향이 일치. 사람이 확인 후 진행.
   - **Partial Consensus** — 공통점과 차이점 공존. 차이점 요약 후 사람이 판단.
   - **Divergent** — 방향 불일치. 사람 필수 개입, 요구사항 재정의 검토.

4. 다음 형식으로 출력한다:

```

{에이전트 A 응답}

━━━ 💻 시니어 개발자 ━━━━━━━━━━━━━━━━━━━━━━━
{에이전트 B 응답}

━━━ 🔒 QA/보안 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
{에이전트 C 응답}

━━━ 📊 합의 분석 결과 ━━━━━━━━━━━━━━━━━━━━━━
  합의 수준: {Strong Consensus | Partial Consensus | Divergent}

  공통점:
```

5. 결과를 `docs/verifications/{date}-{slug}.md`에 다음 형식으로 저장한다:

```markdown

> 날짜: {YYYY-MM-DD}
> 합의 수준: {Strong Consensus | Partial Consensus | Divergent}

## 질문
{전체 질문}

## 아키텍트
{에이전트 A 응답}

{에이전트 B 응답}

## QA/보안
{에이전트 C 응답}

- **합의 수준**: {Strong Consensus | Partial Consensus | Divergent}
```

## 합의 분석 후 행동 제안

- **Strong Consensus** → "핵심 방향이 일치합니다. 확인 후 진행하시죠."
- **Partial Consensus** → 차이점을 요약하고 사용자에게 판단을 요청
- **Divergent** → "관점이 크게 다릅니다. 질문을 다시 정의하거나 사람이 직접 판단해야 합니다."

## 모드 판별 기준

1. `--agent` 옵션이 있으면 → Mode B
2. MCP 도구 `x_verify`가 사용 불가능하면 → Mode B
3. 그 외 → Mode A (MCP 도구 `x_verify` 사용)

## 합의 분석 후 행동 제안 (Mode A)

Mode A의 합의율 기반:
- **90%+** → 자동 채택 권고
- **70~89%** → AI가 차이점 요약, 사람 판단
- **70% 미만** → 사람 필수 개입, 질문 재정의 검토
- **AI 1개 (AGENT REVIEW)** → 에이전트 자신의 의견과 비교하여 판단 제시

# Notes
- Mode A 결과는 자동으로 `docs/verifications/`에 저장됨
- `--no-save` 옵션으로 저장 없이 실행 가능
- `--agent` 옵션으로 API 키가 있어도 에이전트 모드 강제 사용 가능
- `--claude`, `--gpt`, `--gemini` 옵션으로 특정 AI만 호출 가능 (Mode A)
- 설계 판단, 아키텍처 선택, 기술 스택 결정 등에 사용
- `x_verify` MCP 도구가 사용 불가능한 환경에서는 자동으로 Mode B(에이전트 모드)로 전환한다.
